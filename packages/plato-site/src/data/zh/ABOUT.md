# 关于Plato

> *“他不仅是最善良的人，也是最智慧的人。”*

**柏拉图 (Plato)是一个伪装成解谜游戏的证明助手。** 它的名字来源于那位坚信真知源于推导、而非被动告知的哲学家。

我建造它，是因为受够了逻辑学在课本里枯燥死板的模样。大多数证明助手令人望而生畏：它们说着晦涩的语法，要求先啃完几百页文档，默认你已经拥有博士学位。而柏拉图想换个方向：当你输入 `(assume A)`，看到一行新证明跳出来，心里一动，再打几行——不知不觉间，你其实已经在用蕴含式进行严丝合缝的思考了。

它不打算教你逻辑学的概念，它只想让你真正去“做”逻辑。最好的学习方法永远是亲自动手、犯错，然后慢慢培养出对证明的直觉与手感。

## 如何使用

你会看到一个待证明的逻辑命题。在基于 S-表达式的 REPL 终端中输入命令——如 `(assume A)`、`(->-elim 2 1)`、`(and-intro 1 3)`——核心引擎会根据自然演绎规则逐行检查你的推导。

前几关会有引导卡片陪你走完最初的证明，提示系统也会在你卡住时提供帮助。每解出一道题都会解锁新的战术规则，一步一步筑起你的逻辑工具箱。

目前系统内包含 **61 道题目**，分为三个章节——命题逻辑（38题）、一阶逻辑（15题）和模态逻辑/K系统（8题）。每个章节都以一段**探索对话**开场——由历史上的逻辑学家们（柏拉图与亚里士多德、德摩根与弗雷格、亚里士多德与大师）进行简短的插图式对话，为章节主题定调，然后才让你进入证明环节。随着解题，你将解锁 **24条战术规则**，每一条都配有 LaTeX 推理规则和自然语言描述。

Plato 引擎支持两套**公理集**——PL 和 FOL——在导航栏中以彩色标签显示（FOL 开启为绿色，FOL 关闭为橙色）。为什么要分开？见下文。

你也可以通过 URL 或 JSON 文件**加载自定义题目**——引擎不在乎目标是从哪里来的。

## 为什么需要两套公理集？

Plato 位于三个逻辑系统的交汇处，而它们之间的互动方式——或者说，拒绝互动的方式——本身就是这个设计有趣的地方。

**命题逻辑（PL）** 是三者中最简单的。你对由 $\lnot$、$\land$、$\lor$、$\to$、$\top$、$\bot$ 连接的原子命题进行推理。它是可判定的：一张真值表就能在有限时间内解决任何命题逻辑命题。这是整个体系的基石。

**一阶逻辑（FOL）** 加入了量词（$\forall$、$\exists$），让你能够讨论对象及其属性——"所有人都会死"，"苏格拉底是人"。FOL 是半可判定的：如果一个公式是有效的，就一定有证明；但如果它无效，没有任何算法总能告诉你答案。不过，对于实际的自然演绎，FOL 行为良好且已被彻底研究透彻。

**模态逻辑（K系统）** 在命题逻辑之上加入了 $\Box$（必然）和 $\Diamond$（可能）。K系统是可判定的——事实上，你可以将其翻译为一阶逻辑的一个片段（所谓*标准翻译*，转换为可能世界语义），因此它并不比 FOL 更危险。

麻烦在于当你试图把它们组合在一起。**一阶模态逻辑（FOML）**——FOL 加上 $\Box$ 和 $\Diamond$——完全是另一回事。量词和模态算子并不友好相处。考虑巴坎公式（Barcan formula）：

$$
\forall x \Box P(x) \to \Box \forall x P(x)
$$

这个公式应该成立吗？这取决于你的立场。如果你认为对象在跨越可能世界时可以凭空出现或消失（可变域），答案是否定的。如果你相信所有可能对象的集合是固定不变的（常域），答案是肯定的。这里不存在唯一的标准语义——关于"存在"在不同可能世界中意味着什么，不同的哲学承诺会引出不可调和的不同逻辑。更糟糕的是，即使是 FOML 的一元片段（只有一元谓词）也是不可判定的——相比之下，一元 FOL 却是可判定的。量词和模态之间的互动如此丰富，以至于将系统推到了任何有限算法都无法驯服的疆域之外。

Plato 的解决方案很朴素：**不把它们组合在一起。**命题逻辑和模态逻辑章节在 PL 下运行（你可以使用 $\Box$ 和 $\Diamond$，但不能使用量词）。一阶逻辑章节在 FOL 下运行（你可以使用 $\forall$ 和 $\exists$，但不能使用模态算子）。每个系统都待在自己可判定的领域内，导航栏中的公理集标签让你一眼就知道此刻手上握着哪些工具。

## 为什么是自然演绎？

因为自然演绎模仿的是人类真实的推理方式。

大多数逻辑课最喜欢考真值表。一行一行填，一列一列查。它确实死板有效，但那感觉纯粹是在机械搬砖，不像是在*思考*。

自然演绎完全不同：你做出假设、推导结论、消去假设。每一步都对应着你脑子里的真实推理动作——*“如果我知道 p，那我就知道 q。”* **你不是在计算真假，你是在构建论据。**

## 关于难度

自然演绎确实很难。构建证明没有一劳永逸的通关算法，你必须自己去摸索“手感”。

看起来只有一行的题目可能极具挑战性。卡关了极其正常，放它一马，去洗个澡或者睡一觉，第二天回来瞅一眼，往往就豁然开朗了。**这不是一个可以速通的新手教程，而是一种需要日积月累的脑力修行。**

## 技术实现

Plato 的底层是一个典型的 pnpm monorepo：

* **核心引擎 (`plato-lib`)**：纯手工打造的 Rust 形式化验证引擎，通过 `wasm-pack` 编译为 WebAssembly 运行在前端。公式、上下文、断言、推理规则、S-表达式解析器全都在这，包含约 3000 行 Rust 代码。
* **用户界面 (`plato-site`)**：基于 Vue 3 + TypeScript + Vite 的单页应用，使用 vue-router（hash 模式）驱动页面切换，搭配过渡动画。
* **渲染系统**：KaTeX 负责数学公式的实时渲染，其余一切文本使用 JetBrains Mono 字体。
* **国际化**：完整的中英文双语支持（i18n），并附带了逻辑学术语对照表以及中文自然语言生成层，用于证明步骤的解释输出。
